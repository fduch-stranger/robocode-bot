# Bot Architecture

All bots share reusable logic from `bots/bot_core/`. Prefer changing shared code when behavior is genuinely common; keep strategy/personality in the bot directory.

Canonical docs:
- `docs/bot-shared-systems.md`: common control loop, target cache, radar, virtual guns, gun learning, fire gate, enemy fire detection, movement learning, minimum risk movement, telemetry semantics.
- `docs/bot-core-data-structures.md`: target snapshots, gun waves, movement waves, rolling KNN memory, guess-factor profiles, stats buffers, enemy fire-power prediction, telemetry records, approximation tradeoffs.
- Bot-specific READMEs: bot state machines, movement mode choices, firepower policy, gun policy, key telemetry, tuning checklist.

Current bot roles:
- Adaptive Prime: 1v1-first champion candidate. Uses go-to surfing when surfable waves exist, potential-field movement as fallback, minimum-risk in melee, and aggressive confidence/energy-based firepower.
- Chase Lock: pressure/chase bot. More target-lock oriented and can be weak if it tunnels stale targets or prioritizes pressure over dodge.
- Circle Strafer: stable defensive orbital bot with wall escape, separation, and conservative fire.
- Sweep Pressure: direct sweeping pressure bot with projected wall avoidance and steady close-range fire.

Gun architecture context:
- `VirtualGunSystem` is the shared gun facade. It builds `AimContext`, gathers `GunBearing` values from a `GunRegistry` of concrete gun components under `bot_core.gun.guns`, handles wave/scoring orchestration, and publishes production `GunVisit` records back to components for learning. `DynamicClusterGun` owns `RollingKnnBuffer`; Traditional GF and anti-surfer own separate profile data. `AimModeSelector` owns sticky virtual-gun selection and can report `GunSwitchCandidate` diagnostics through `select_with_diagnostics()` using per-mode policy data rather than mode-specific selector branches.
- `bot_core.gun.should_log_switch_decision()` centralizes sampled switch-decision telemetry gating for mode changes and blocked better-scoring candidates.
- `VirtualGunSystem.clear_round_state()` is round-scoped: it clears transient waves/history/component round state, but keeps active selected gun modes, production score stats, and component learners so multi-round battles continue learned selector behavior.
- `VirtualGunSystem.clear_battle_state()` and `MovementFlattener.clear_battle_state()` are game-scoped: bot `on_game_started` handlers use them to clear opponent-id-scoped learning when the GUI/server starts a new game, because numeric target ids can be reused for different opponents after bot composition changes.
- `gun.switch_decision` telemetry records sampled selector decisions/rejections: unavailable, visits, score floor, margin, superseded, selected, current, and forced.
- `GunRuntimeConfig` is the bot wiring boundary. Concrete gun tuning lives in each component package config (`DynamicClusterGunConfig`, `TraditionalGfGunConfig`, etc.); the root `bot_core.gun` package exports shared contracts only. `GunSelectorConfig.switch_confidence_visits`/`switch_confidence_penalty` optionally apply a low-visit uncertainty penalty before switch gates. In `gun.switch_decision`, `score` is the adjusted decision score and `raw_score` is the unadjusted virtual-gun score.
- `gun.eval_wave_visit` is optional neutral evaluation telemetry. Eval-wave stats are separate from production switching stats and should be used for diagnostics, not A/B performance gates.
- `tools/gun_eval_summary.py` reports calibration by target and gun mode: switch-time score/visits, next-N real post-switch hit rate, production wave average, eval-wave average, and score-vs-hit gaps. Use eval gaps as diagnostic evidence only.
- Adaptive `traditional_gf` is suspected to be under-modeled, not merely mis-thresholded. Check active plans in `docs/plans/README.md`, especially selector/model calibration work, before changing its gates again.
- Traditional GF shared defaults enable segmented profile recording when a bot provides 1v1 segment keys: exact and coarse segment activation use `12/48`, the coarse key is fixed to distance, lateral speed, and wall margin, peak selection defaults to density-supported, and lower-trust sources are centered (`global=0.8`, `coarse=0.7`, `coarse_blend=0.8`) to reduce positive GF over-aiming. Source-bias correction remains implemented but disabled by default.
- Alternate Traditional GF coarse-key env controls were removed after July 2026 BasicGFSurfer validation did not justify the extra tuning surface.
- A July 2026 BasicGFSurfer 12-round matrix found no default promotion for alternate coarse-key shapes. Forced `distance_lateral_velocity_age` had the best Traditional GF hit rate/error, `distance_lateral_advancing` scored well in forced mode, but normal selector results stayed mixed. Use `tools/gun_eval_summary.py` source-level real hit, profile diagnostics, and GF error summaries before changing source penalties or centering factors again.
- A focused July 2026 24-round BasicGFSurfer follow-up did not confirm `distance_lateral_velocity_age`: forced default `distance_lateral_wall` beat it on score, Traditional GF hit rate, and selected GF abs error. Normal `velocity_age` scored better overall but fired far fewer Traditional GF shots with weaker hit rate/post-switch evidence, so do not promote it or make selector calibration trust it more from that data.
- A July 2026 forced Traditional GF local matrix found `traditional_gf_min_samples=12` improved over the previous 28-sample warmup: Chase 12316/72 firsts, Circle 11399/72, Sweep 11618/72, total 35333/216 versus prior forced-GF total 34769/214. Earlier source-centering and disabling-exact-segment sweeps against repo bots did not promote from that matrix alone; the later BasicGFSurfer telemetry pass is the current reason for the shared lower-trust source-centering defaults. A per-target/source GF residual-bias learner was added as an experimental hook with telemetry fields, but shared defaults keep `source_bias_max_correction=0.0` because Chase checks with max correction 0.22 scored 12084/71 and max correction 0.10 scored 12291/72, both below the 12316/72 min-samples baseline.
- `track` telemetry may include `traditional_gf_*` diagnostics for global/segment GF peaks, profile weights, blend, selected GF, and source.
- `gun.traditional_gf_profile` emits sampled traditional-GF diagnostics independently from switch decisions, so scripted and forced-gun runs can inspect the model even when selector or `track` samples are sparse.
- Gun selection now uses `GunModeTraits` in each component's `GunModePolicy`: `dynamic_cluster` is labeled as the primary KNN-GF learner, `traditional_gf` as a situational profile-GF gun, `linear`/`head_on` as fallbacks, and force-testable guns such as `anti_surfer`/`displacement` as situational. `AimModeSelector` combines raw virtual score, confidence/source penalties, context-aware gates, and a `decision_bonus` from traits plus generic decision context. `AimContext.movement_tags` are derived from recent target path curvature, speed stability, and current lateral motion; Dynamic Cluster exposes nonlinear/adaptive/surfer history tags as selector decision context, linear matches low-lateral/stable movement, and Traditional GF can combine stable-pattern history with trusted source tags. The selector also supports role-aware margins: primary KNN can leave fallback linear earlier, situational guns need a larger normal margin over primary KNN, and that larger margin relaxes only when primary KNN has enough low-score visits and the situational candidate has source/context evidence. Situational current guns with global-only degraded source lose their normal sticky score advantage, but replacement candidates must still pass visit and score gates; `gun.switch_decision` can report `reason=source_degraded`. Optional eval waves stay out of production `GunStats` and concrete learners, but the selector can use the separate eval scorer for capped `eval_score_bonus` and partial `effective_visits` credit after enough eval visits. `gun.switch_decision` reports `decision_bonus`, `current_decision_bonus`, `eval_score_bonus`, `eval_visits`, and `effective_visits`.
- Repo bots now use aligned aggressive live-gun gates via `bot_core.gun.policy.SHARED_GUN_POLICY_DEFAULTS` and `selector_config_from_policy()`: KNN warmup is lowered (`knn_min_samples=30`, base switch visits `12`, score floor `0.03`), and Traditional GF is optional/situational (`45` visits, `0.10` score floor) instead of a broadly preferred late gun. Adaptive overrides the shared switch margin to `0.035` and keeps additional confidence-penalty tuning. Adaptive still has nested `TraditionalGfPolicy` source-aware gates: global source stays more conservative, trusted exact/coarse source can promote earliest, and blended sources interpolate by blend weight.
- Chase Lock, Circle Strafer, and Sweep Pressure each keep bot-local `GunPolicy`, fire, target, radar, and movement policy surfaces in `chase_config.py`, `circle_config.py`, and `sweep_config.py`, but their live gun gates are aligned with the shared KNN-first experimental selector policy. Use switch-decision, eval-wave telemetry, and forced-gun A/Bs before promoting further gate or trait changes.
- Adaptive live-selects linear, traditional GF, and dynamic cluster. `anti_surfer` and `displacement` remain force-testable for Adaptive but are not normal live-selectable modes. Chase/Circle/Sweep live-select linear, traditional GF, and dynamic cluster.
- Force env vars: `ROBOCODE_ADAPTIVE_GUN_MODE`, `ROBOCODE_CHASE_GUN_MODE`, `ROBOCODE_CIRCLE_GUN_MODE`, and `ROBOCODE_SWEEP_GUN_MODE`.
- Eval-wave env vars: `ROBOCODE_ADAPTIVE_GUN_EVAL`, `ROBOCODE_CHASE_GUN_EVAL`, `ROBOCODE_CIRCLE_GUN_EVAL`, `ROBOCODE_SWEEP_GUN_EVAL`, plus matching `_INTERVAL` vars.

Telemetry event examples: `track`, `gun.switch`, `gun.switch_decision`, `gun.wave_visit`, `gun.eval_wave_visit`, `gun.fire_drift`, `enemy.fire_detected`, `enemy.gun_heat_wave`, `movement.profile_visit`, `movement.flatten`, `movement.minimum_risk`, `bullet.fired`, `bullet.hit_bot`, `hit.bullet`.

Engine-assumption notes:
- Shared movement prediction follows Tank Royale target-speed order: update speed, move on previous direction, turn by new-speed turn limit, line-preserving wall clip, and zero speed after a wall hit.
- `distance_remaining` in the predictor is a local go-to/surf planning approximation; Tank Royale server intent processing is target-speed based.
- Bullet shadows should be recorded from actual `BulletFiredEvent.bullet` state, and `gun.fire_drift` audits production wave source/direction/power/speed against that event.

Do not duplicate formulas across docs. Keep exact math in `docs/bot-core-data-structures.md`; overview docs should link there.
