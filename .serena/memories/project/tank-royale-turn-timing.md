# Tank Royale Turn Timing

As of upstream Tank Royale 1.0.2, default `GameSetup.turnTimeout` is `DEFAULT_TURN_TIMEOUT_MICROS = 5_000` microseconds. Default TPS is 30, but TPS is not the bot compute budget; the server schedules turn processing by `turnTimeout` and applies visual/TPS delay after processing. Local `tools/battle-runner` uses `BattleSetup.oneVsOne`/`melee` without overriding turn timeout, so local battles use the engine default unless the runner is changed. In local CLI smoke runs on 2026-07-05, Python `Bot.turn_timeout` reported `30_000` us; treat the runtime property as source of truth for warning thresholds.

Python Bot API flow: server sends `TickEventForBot`; the bot thread runs event handlers and local decision code; `go()` dispatches queued events, sends `BotIntent`, then waits for the next tick. The analysis budget is everything between tick receipt and sending intent inside `go()`. If the bot misses the deadline, `SkippedTurnEvent` is sent for the previous turn. Server-side `ModelUpdater` keeps a persistent `botIntentsMap` and merges only newly received fields, so a skipped turn effectively repeats the last stored commands rather than creating a fresh zero intent.

Current project instrumentation: shared `TurnTimingTelemetry` emits sampled `bot.turn_timing` and immediate `bot.skipped_turn` records for main local bots and the BasicGFSurfer port. Severity scales to the runtime `turn_timeout` (`warn` at 50%, `danger` at 80%, `critical` at 10% remaining) with fixed minimum thresholds for missing timeout data. `tools/intent_gap_summary.py` summarizes missing/duplicate intent turns from `--intent-diagnostics`. Smoke runs on 2026-07-05 showed continuous intent turns and only `ok` timing samples for Adaptive vs Chase and Adaptive vs BasicGFSurfer Port.
